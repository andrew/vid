# VID in brief

Finding vulnerabilities in open source used to be slow work, and the disclosure process built around it (private report to the maintainer, wait for a fix, request a CVE, publish an advisory) was sized for that pace. Automated tools, particularly LLM-based scanners, now surface real bugs in minutes, and anyone running them against popular libraries for a few days accumulates more findings than they could responsibly disclose in a year. The same libraries are being scanned by many people at once, so a volunteer maintainer receives the same report five times over and eventually stops reading security mail. Users of the library see nothing until a CVE turns up in their scanner months later.

A VID is computed rather than issued. Take the package's purl, a standard string like `pkg:npm/lodash`, and the SHA-256 gitoid of the file where the bug lives, concatenate them in a fixed way, and hash the result into a short string like `VID-1-4fta-nqe5-ppht-2m6r-2tpk-o57c`. Because the inputs describe where the vulnerability is rather than who found it, two researchers looking at the same bug compute the same identifier without talking to each other or to any authority. The CWE, severity, line number, and prose analysis travel as metadata alongside the identifier; they are not hashed in, because two people will agree on which file far more reliably than on which CWE.

A maintainer receiving five reports that each carry a VID can see at a glance that four are duplicates. A researcher can publish a VID the day they find something as a timestamped commitment that reveals nothing on its own; when the advisory goes out the inputs are disclosed and the early date is provable. A consumer holding a set of VIDs can hash every file in their installed dependency tree, learn which packages to examine before any CVE exists, and record their reachability conclusion against the VID so the analysis is already done when their scanner eventually fires. None of this replaces CVE: the VID aliases to the CVE once one is assigned, there is no required database, and a VID can be computed for a false positive as easily as for a real bug, so it names a claim rather than vouching for one.

The v1 construction hashes the whole sink file, so an unrelated edit anywhere in that file produces a new identifier for the same unfixed bug. Two branches explore narrowing the hashed unit to the function containing the sink, located by tree-sitter using the grammars packaged with `github.com/git-pkgs/outline`. The `vid2-function-bytes` branch gitoids the raw byte range of the enclosing function node, so the grammar only decides where the function starts and ends and the hash itself is plain source. The `vid2-function-ast` branch hashes a canonical S-expression of the function's syntax tree with comments and whitespace dropped, pinning the grammar version into the preimage so two parties converge only when they share a grammar release. On the worked example four still-vulnerable releases collapse to one identifier under the AST variant, two under the bytes variant, three under v1; [COMPARISON.md](COMPARISON.md) has the full table and a recommendation.
